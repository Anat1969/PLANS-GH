# אבטחת ה-Supabase של PLANS-GH — מדריך הפעלה

## המצב היום (חשוב להבין)

כל האפליקציה עובדת עם **מפתח ציבורי אחד** (`sb_publishable_...` בתוך index.html). המשמעות:
כל מי שפותח את קוד הדף יכול לקרוא **ולכתוב** לכל טבלה. מערכת התפקידים במסך
(אדריכלית/הנהלה/יזם) אוכפת התנהגות בממשק בלבד — לא בשרת.

לתיקון מלא צריך שני דברים ביחד: RLS בשרת (הקבצים כאן) + כניסה אמיתית בקליינט
(Supabase Auth). בגלל זה הערכה מפוצלת לשני שלבים.

## שלב א' — להריץ עכשיו (5 דקות, לא שובר כלום)

1. היכנסו ל-[Supabase Dashboard](https://supabase.com/dashboard) → הפרויקט `slcpldoaaagkoozpbjsk`.
2. **SQL Editor** → New query → הדביקו את כל התוכן של `migrations/00_phase_a_baseline.sql` → **Run**.
3. מה קיבלתם:
   - אילוצי הייחודיות שהסנכרון של האפליקציה מניח (מונע כשלי upsert עתידיים).
   - **חסימת מחיקה** של רשומות תוכניות ושל שורות ההגדרות — גם מי שמשיג את המפתח
     הציבורי לא יכול למחוק את המאגר (הכתיבה עדיין פתוחה — זו מגבלת שלב א').
   - שאילתת ביקורת בסוף — ודאו שאין policy זר עם `qual=true` על טבלאות אחרות.

## שלב ב' — הרשאות אמיתיות (ספרינט נפרד, ~יום עבודה)

**אל תריצו את `01_phase_b_rls.sql` לפני שהקליינט הוסב ל-Auth** — האפליקציה הנוכחית
תפסיק לשמור נתונים ברגע שהכתיבה האנונימית נחסמת.

סדר הספרינט:
1. Dashboard → Authentication: יצירת משתמשים (אימייל+סיסמה) לענת, למנכ"ל ולכל יזם.
2. הרצת `01_phase_b_rls.sql` — יוצר `profiles` + `plan_access` + כל המדיניות.
3. מילוי `profiles.role` לכל משתמש, ושיוך יזמים לתוכניות ב-`plan_access`
   (`plan_key` בפורמט `<tik>__<planNum>`, למשל `1714__`).
4. בקליינט: החלפת מסך הכניסה הנוכחי ב-`supabase.auth.signInWithPassword`,
   ושליחת ה-JWT בכותרות במקום המפתח הציבורי לבדו. (שינוי מרוכז באזור
   SB_HEADERS + loginSubmit — כ-100 שורות.)
5. בדיקות התחזות לפי המטריצה שבסוף קובץ שלב ב' (psql עם rollback — בטוח מול ייצור).

## כלל אצבע

עד שלב ב': קודי הגישה במסך הם נעילת-נימוס, לא אבטחה. אל תכניסו למערכת מידע
שאסור שגורם עוין עם ידע טכני יראה או ישנה.
